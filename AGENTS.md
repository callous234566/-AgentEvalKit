# 项目开发规则

## 项目概述

本项目是个人 RAG 知识库助手，技术栈为 SiliconFlow + LangChain + Chroma + FastAPI + Streamlit。目标是提供稳定、可维护、支持中文知识库名称和本地持久化的个人文档问答系统。

## 依赖管理规则

1. 安装新库前必须执行兼容性分析：
   - 识别当前核心依赖版本，例如 Python、LangChain、Chroma、FastAPI、Streamlit。
   - 判断是否与现有依赖冲突。
   - 优先采用最小变更策略。
   - 明确说明变更原因、推荐版本和影响范围。
2. `requirements.txt` 必须同步维护：
   - 新增依赖必须写入 `requirements.txt`。
   - 版本号使用 `>=`，除非有明确兼容性风险。
   - 按功能模块分组注释。
3. 禁止操作：
   - 禁止批量升级整个环境。
   - 禁止安装未经验证的预发布版本。
   - 禁止忽略依赖冲突警告。

## 代码修改规则

1. 最小改动原则：
   - 修复问题时不重构无关代码。
   - 新增功能时复用现有模块。
   - 保持代码风格一致。
2. RAG 链路修改注意事项：
   - 文档解析模块变更后，需要验证 PDF、TXT、DOCX、Markdown。
   - 文本分块参数变更后，需要评估检索效果。
   - 向量存储变更后，需要考虑数据迁移和缓存失效。
   - 模型配置变更后，需要验证输入输出格式。
   - 检索缓存变更后，需要验证上传、删除、重命名后的缓存失效。
3. 错误处理规范：
   - 禁止通过删除核心逻辑、跳过校验、吞掉异常来伪装修复。
   - 所有异常必须记录日志。
   - API 层必须返回用户友好的错误信息。

## 服务稳定性规则

1. 启动模式：
   - 生产或长期运行环境必须禁用 `--reload`。
   - 推荐使用 `--workers 1 --timeout-keep-alive 120`。
   - 优先使用 `start_backend_stable.ps1`。
2. 外部依赖容错：
   - API 调用必须设置超时。
   - 多模态、reranker 等非核心增强能力失败时，应降级而不是中断主流程。
   - 启动时预加载关键资源，减少首次请求超时。
3. 异常防护：
   - 必须设置全局异常捕获。
   - 未捕获异常必须写入日志。
   - 不允许静默失败。

## 数据持久化规则

1. 名称映射持久化：
   - 用户可见名称与 Chroma 内部名称必须保存到磁盘。
   - 使用 JSON 文件，例如 `collection_name_mapping.json`。
   - 服务启动时自动加载，变更时立即保存。
2. 状态同步：
   - 前端状态变更必须与后端数据同步。
   - 创建、删除、重命名资源必须调用 API。
   - 不允许只修改前端状态而不修改后端数据。

## 用户体验规则

1. 中文支持：
   - 界面语言统一为中文。
   - 知识库名称支持中文显示和持久化。
   - Streamlit 默认英文提示可以通过 CSS 清理。
2. 界面优化：
   - 保持元素对齐和布局清晰。
   - 输入框、按钮、反馈提示风格统一。
   - 操作结果必须及时反馈成功或失败。

## 测试验证规则

修改后至少验证受影响路径：

- 文档上传是否正常。
- 文本分块是否符合预期。
- 向量检索是否返回结果。
- 问答流程是否完整。
- 服务启动是否稳定。
- 中文知识库名称是否正确显示。
- 缓存是否在上传、删除、重命名后失效。

环境变量变更必须更新 `.env.example`。

## 文档记录规则

1. 每次修改后更新 `BUG_LOG.md`。
2. 重复出现的问题上升为长期规则写入 `AGENTS.md`。
3. 接口变更同步更新 `README.md`。
4. 技术实现细节更新到 `RAG技术详解.md`。
5. 维护路线和未完成项更新到 `项目维护手册.md`。

## 已上升为长期规则的问题

- 不得在真实 `chroma_db/collection_name_mapping.json` 上做临时测试；测试名称映射必须使用临时目录。
- 清理历史编码问题后，必须运行 `py_compile`，防止乱码掩盖的代码粘连问题。
- 检索链路中的全量 collection 文本读取应统一走快照缓存，除非需要 Chroma ids。
- 新增运行时缓存时，必须设置上限并明确缓存 key 覆盖哪些行为配置。
- 使用 SiliconFlow Embedding 兼容接口时，`OpenAIEmbeddings` 必须设置 `check_embedding_ctx_length=False`，避免 LangChain 将中文文本转换为兼容接口不接受的 token-id 数组；文本长度保护统一由项目分块逻辑负责。
- 上传队列删除必须以 Streamlit session state 为准，不得依赖隐藏上传器 DOM 结构；新增文件时必须保留已移除记录和已完成任务状态，成功文件不得被主处理按钮重复入库。
- 对话生成期间仅在用户仍靠近底部时自动跟随最新回答；用户主动查看历史后不得强制跳转，必须提供可见的“查看最新回答”入口恢复跟随。
- 移动端侧边栏作为覆盖抽屉显示时，必须保留 Streamlit 原生收起控件的可见性和点击能力；固定输入框必须适配窄屏与底部安全区，不能遮挡页面内容。
- 前端页面级反馈不得直接使用 Streamlit 默认 `st.info`、`st.warning`、`st.success`、`st.error` 大色块；需要占位提示时使用统一 `render_status_notice()`，瞬时成功反馈优先使用 `st.toast()`。
- 前端小步样式调整后通常不需要重启服务；用户会刷新界面验证，失败后再反馈。只有依赖、启动配置、后端接口或运行时代码路径变更时才考虑重启。
- Streamlit 前端优化必须优先保持现有稳定交互不回退，尤其是侧边栏三点菜单、上传队列删除、失败文件移除、固定输入框、最新回答入口和深浅色切换。
- 深色模式新增或调整组件时，必须同步检查最终深色覆盖层，避免亮白、亮红硬编码在深色模式下破坏界面。
- CSS 维护只清理确认无调用的孤儿规则或明确重复覆盖；不得为了降低体积而大拆当前可用结构、改动按钮 key、session state key 或 DOM hook。
- 前端契约类回归应优先运行 `tests/test_frontend_ui_contracts.py`，防止默认 Streamlit 大色块、默认进度条、旧上传列表、旧侧边栏菜单等回潮。
- PowerShell 直接读取中文文件可能显示乱码；判断文件是否真实乱码时，优先用 Python `Path(...).read_text(encoding="utf-8")` 检查。

## Codex 项目理解入口

- 开始复杂任务前，优先阅读 `PROJECT_MAP.md` 了解后端、前端、RAG、测试和文档入口。
- 排查历史坑和待确认风险时，优先阅读 `docs/KNOWN_ISSUES.md`，不要只凭记忆修复。
- 遇到项目术语不确定时，查看 `docs/GLOSSARY.md`。
- 需要给 Codex 下发重复任务时，优先复用 `docs/codex-prompts/frontend-fix.md`、`docs/codex-prompts/backend-debug.md`、`docs/codex-prompts/rag-debug.md`。
- 项目级 skills 位于 `.agents/skills/`；通用个人 skills 位于 `$HOME/.agents/skills/`。项目规则优先级高于通用流程。
